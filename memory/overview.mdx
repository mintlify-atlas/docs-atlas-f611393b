---
title: "Memory system overview: four layers, one portable brain"
description: "How the four-layer memory architecture stores, scores, and retrieves context across sessions so the agent compounds knowledge over time."
---

The memory system is the portable brain inside `.agent/memory/`. It divides knowledge into four layers with distinct retention policies: volatile task state lives in `working/`, raw experience logs live in `episodic/`, distilled reusable patterns live in `semantic/`, and user-specific preferences live in `personal/`. This separation lets the agent keep ephemeral state close to hand without polluting long-term knowledge, and ensures personal conventions never accidentally bleed into shared lessons.

## Memory layers

<Note>
At session start the agent reads memory in the order shown below. Reading personal preferences first means user conventions override any conflicting defaults before any work begins.
</Note>

| Layer | Volatility | Key files | Purpose |
|-------|-----------|-----------|---------|
| `working/` | High — archived after 2 days | `WORKSPACE.md`, `REVIEW_QUEUE.md` | Live task state and pending candidate backlog |
| `episodic/` | Medium — decays after 90 days (salience < 2.0) | `AGENT_LEARNINGS.jsonl` | Raw experience log, scored by salience |
| `semantic/` | Permanent — never deleted | `LESSONS.md`, `DECISIONS.md` | Distilled patterns and architectural choices |
| `personal/` | Permanent — never merged | `PREFERENCES.md` | User-specific conventions |

### working/ — live task state

`WORKSPACE.md` holds the current task breakdown, open questions, and progress notes. It is the scratchpad for an in-progress session. `REVIEW_QUEUE.md` is generated automatically by `auto_dream.py` and surfaces how many candidate lessons are waiting for review. Both files are volatile: `archive.py` moves `WORKSPACE.md` to `episodic/snapshots/` if it has not been updated in 2 days.

```
.agent/memory/working/
├── WORKSPACE.md      ← current task state; updated throughout the session
└── REVIEW_QUEUE.md   ← generated summary of staged candidates
```

### episodic/ — raw experience log

`AGENT_LEARNINGS.jsonl` is an append-only log written after every significant action via `memory_reflect.py`. Each entry is a JSON object with fields for `action`, `reflection`, `detail`, `skill`, `timestamp`, `salience`, and optionally `recurrence_count`. The agent reads the top-k entries by salience score, not by recency — a single critical failure from three months ago outranks dozens of routine successes from last week.

Entries older than 90 days with a salience score below 2.0 are moved to `episodic/snapshots/` by `decay.py` during the nightly dream cycle. Git history retains every archived entry indefinitely.

```
.agent/memory/episodic/
└── AGENT_LEARNINGS.jsonl   ← raw experience; JSONL, one entry per line
```

### semantic/ — distilled patterns

`LESSONS.md` is the rendered view of `semantic/lessons.jsonl`, which is the append-only source of truth for accepted lessons. **Never hand-edit `LESSONS.md`** — it is regenerated by `render_lessons.py` every time a candidate is graduated or rejected. Hand-curated content written above the sentinel line is preserved across every render.

`DECISIONS.md` records past architectural choices in a freeform format. Unlike lessons, decisions are written directly; there is no staging or review pipeline for them.

```
.agent/memory/semantic/
├── LESSONS.md          ← rendered view; do not edit directly
├── lessons.jsonl       ← source of truth for graduated lessons
└── DECISIONS.md        ← architectural choices; hand-maintained
```

### personal/ — user preferences

`PREFERENCES.md` is populated by the onboarding wizard and is the first file read at every session start. It holds stable user conventions — name, language, explanation style, test strategy, commit message format, code review depth — that should apply to every interaction. This layer is isolated by design: its contents never flow into `semantic/` and cannot be overwritten by the clustering pipeline.

```
.agent/memory/personal/
└── PREFERENCES.md    ← populated by onboarding wizard; edit any time
```

## Read order

AGENTS.md specifies the exact sequence in which the agent loads memory at session start:

```
1. memory/personal/PREFERENCES.md      ← user conventions (highest priority)
2. memory/working/WORKSPACE.md         ← current task state
3. memory/working/REVIEW_QUEUE.md      ← pending candidates
4. memory/semantic/DECISIONS.md        ← architectural choices
5. memory/semantic/LESSONS.md          ← distilled patterns
6. memory/episodic/AGENT_LEARNINGS.jsonl  ← top-k by salience
```

Reading in this order ensures personal preferences establish context before any task state, and task state is known before the agent consults general knowledge.

## Query-aware retrieval

Not all episodic entries load into context on every session. `AGENT_LEARNINGS.jsonl` may grow large; the agent selects the top-k entries using a **salience × relevance** score. Salience is computed by `harness/salience.py` and factors in outcome severity, recurrence count, and confidence. Relevance is the Jaccard similarity between the current task's word set and the episode's feature set.

This means a recurring pattern with moderate salience but high relevance to today's task will outrank a high-salience episode from an unrelated context. The nightly dream cycle (`auto_dream.py`) also uses this scoring — only episodes whose `canonical_salience` clears a threshold of 7.0 are eligible for staging as candidate lessons.

## Logging to episodic memory

Skills and the host agent log experience entries via `memory_reflect.py`. The tool is the single write path to `AGENT_LEARNINGS.jsonl`; writing directly to the file bypasses salience scoring and the decay index.

```bash
python3 .agent/tools/memory_reflect.py \
  my-skill \
  "deployed to staging" \
  "deployment succeeded after adding the missing env var" \
  --importance 7 \
  --note "Always check .env.example before deploy"
```

```bash
# Log a failure
python3 .agent/tools/memory_reflect.py \
  git-proxy \
  "force-push to main" \
  "blocked by permissions.md" \
  --fail \
  --confidence 0.95
```

The `--importance` flag (1–10) feeds directly into the salience score. Failures automatically receive higher base salience than successes.

## How knowledge compounds

<Steps>
  <Step title="Skills log every action">
    After each significant event, a skill calls `memory_reflect.py` to append an entry to `episodic/AGENT_LEARNINGS.jsonl`.
  </Step>
  <Step title="Nightly clustering stages candidates">
    `auto_dream.py` clusters recurring patterns from the episodic log into candidate lessons and writes them to `memory/candidates/`. Nothing is written to semantic memory yet.
  </Step>
  <Step title="Host agent reviews candidates">
    At the next session, the agent checks `REVIEW_QUEUE.md`. If the backlog is past threshold (> 10 pending or oldest > 7 days), the agent reviews before starting new work.
  </Step>
  <Step title="Graduation appends to lessons.jsonl">
    Accepted candidates are appended to `semantic/lessons.jsonl` and `LESSONS.md` is re-rendered. Rejected candidates retain their full decision history.
  </Step>
  <Step title="Future sessions load relevant lessons">
    At the next session start, the agent reads `LESSONS.md` and retrieves query-relevant episodic entries. Accumulated lessons reduce repeated mistakes without repeating the sessions that produced them.
  </Step>
</Steps>
